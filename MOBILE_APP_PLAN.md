# План: Мобильное приложение POS-кассира для планшетов (Android + iOS)

## Контекст

HookahPOS — SaaS POS-система для кальянных. Сейчас работает только как веб-приложение. Нужно мобильное приложение для планшетов (Android/iPad) с интерфейсом кассира, которое:
- Работает **онлайн и офлайн** (offline-first)
- **Синхронизируется** с сервером при появлении сети
- Поддерживает **печать чеков** на термопринтерах (Bluetooth/USB)

Мобильное приложение реализует **только кассирский функционал** — админские функции (управление товарами, отчёты, настройки) остаются в веб-версии.

| Компонент | Технология | Обоснование |
|---|---|---|
| Фреймворк | **React Native (Expo bare workflow)** | Переиспользование React-парадигмы, Zustand-сторов и бизнес-логики из веб-клиента |
| Локальная БД | **WatermelonDB** | Встроенный sync-протокол (pull/push), ленивые запросы, SQLite под капотом |

| Сеть | **@react-native-community/netinfo** | Детекция онлайн/офлайн |
| Безопасность | **react-native-keychain** | Хранение JWT в Keychain/Keystore |

### Поток данных

**Онлайн:** Действие → Zustand store → API-запрос к серверу → ответ → обновление WatermelonDB → реактивное обновление UI

**Офлайн:** Действие → Zustand store → запись в WatermelonDB (sync_status='pending') → добавление в очередь → обновление UI → при появлении сети → flush очереди → синхронизация

## Серверные изменения

### Миграция 022: Поддержка синхронизации

**Файл:** `server/migrations/022_sync_support.js`

Что добавляется:
1. **`updated_at TIMESTAMPTZ`** на таблицы: `categories`, `products`, `halls`, `tables`, `users`, `orders`, `order_items`, `register_days` (у `guests` уже есть)
2. **`version INTEGER DEFAULT 1`** на таблицу `orders` — оптимистичная блокировка
3. **`device_id VARCHAR(100)`** на таблицу `orders` — отслеживание устройства-источника
4. **Триггер** `update_updated_at_column()` — автоматическое обновление `updated_at` при UPDATE
5. **Индексы** `idx_{table}_updated` — для быстрых delta-запросов
6. **Таблица `devices`** — регистрация мобильных устройств (device_id, tenant_id, platform, push_token)

### Новый роут: `server/routes/sync.js`

| Эндпоинт | Метод | Описание |
|---|---|---|
| `/api/sync/full` | GET | Полный дамп данных тенанта для первоначальной загрузки |
| `/api/sync/delta?since=timestamp` | GET | Инкрементальные изменения с момента timestamp |
| `/api/sync/push` | POST | Загрузка офлайн-очереди (заказы, позиции, оплаты) |
| `/api/sync/device` | POST | Регистрация/обновление устройства |

**Middleware:** `authMiddleware` → `checkSubscription` → handler

### Формат ответа `/api/sync/delta`

```json
{
  "changes": {
    "categories": { "created": [...], "updated": [...], "deleted": [1, 5] },
    "products":   { "created": [...], "updated": [...], "deleted": [] },
    "orders":     { "created": [...], "updated": [...], "deleted": [] }
  },
  "timestamp": 1709913660000
}
```

Soft-delete (`active = false`) передаётся как `deleted` — клиент удаляет из локальной БД.

### Формат запроса `/api/sync/push`

```json
{
  "changes": {
    "orders": {
      "created": [
        { "_tempId": "local-uuid", "table_id": 5, "items": [...], "device_id": "..." }
      ],
      "updated": [
        { "id": 45, "version": 2, "status": "closed", "payment_method": "cash" }
      ]
    }
  },
  "device_id": "device-xyz"
}
```

Ответ возвращает маппинг `_tempId → server_id` и список конфликтов.

### Миграция 023: Фискальные данные

```sql
ALTER TABLE orders ADD COLUMN IF NOT EXISTS fiscal_number VARCHAR(100);
ALTER TABLE orders ADD COLUMN IF NOT EXISTS fiscal_sign VARCHAR(100);
ALTER TABLE orders ADD COLUMN IF NOT EXISTS fiscal_document_number VARCHAR(100);
```

Эти поля заполняются после успешной фискализации чека через ККТ.

### Изменения в существующих роутах

- `server/routes/orders.js` — инкрементировать `version` при каждом обновлении заказа, принимать `device_id`, принимать фискальные данные при закрытии
- `server/socket.js` — добавить событие `sync:available` для уведомления устройств о новых данных

## Протокол синхронизации

### Стратегия разрешения конфликтов

| Данные | Стратегия | Причина |
|---|---|---|
| Товары, категории | Сервер побеждает | Админ меняет на вебе, мобилка только читает |
| Залы, столы | Сервер побеждает | Конфигурация зала — веб-операция |
| Гости | Сервер побеждает | Управление гостями — админская задача |
| Новые заказы (офлайн) | Принимаются оба | Разные устройства не конфликтуют на одной сущности |
| Изменение заказа | Проверка `version` | Если version не совпадает — конфликт, показать пользователю |
| Кассовый день | Сервер побеждает | Критично для финансовой целостности |

### Триггеры синхронизации

1. Запуск приложения — full sync (первый раз) или delta sync
2. Выход из фона — delta sync
3. Восстановление сети — flush офлайн-очереди → delta sync
4. Каждые 30 секунд (онлайн) — delta sync справочных данных
5. WebSocket-события — целевая синхронизация конкретных сущностей
6. Pull-to-refresh — ручной delta sync

## Печать чеков и фискализация (ККТ по 54-ФЗ)

### Два уровня печати

**1. Пре-чеки (нефискальные)** — ESC/POS на обычных термопринтерах:
- Кухонные тикеты для цехов (бар, кухня, кальянная)
- Пре-чеки для гостей (до оплаты)
- Отчёт по смене

**2. Фискальные чеки (ККТ)** — через сертифицированные фискальные регистраторы:
- Чек продажи (при закрытии заказа)
- Чек возврата (при отмене оплаченного заказа)
- X-отчёт / Z-отчёт (открытие/закрытие смены)

### Интеграция с ККТ (АТОЛ / Штрих-М)

| ККТ | SDK (Android) | SDK (iOS) | Протокол |
|---|---|---|---|
| АТОЛ | `ATOL Driver 10` (Java SDK) | `ATOL fprint` (Swift SDK) | Проприетарный через SDK |
| Штрих-М | `JavaPOS Driver` | Через драйвер АТОЛ* | Проприетарный |
| Эвотор | REST API (если Эвотор — сам планшет) | N/A | HTTP localhost |

**Поток фискализации при закрытии заказа:**

```
Кассир нажимает «Оплата» →
  PaymentModal (выбор: наличные/карта/смешанная) →
    fiscal.manager.printReceipt({
      items: [{ name, price, quantity, tax: 'nds20'|'nds10'|'nds0'|'none' }],
      total, payment_method, paid_cash, paid_card
    }) →
      ATOL SDK → ФН (фискальный накопитель) → ОФД (оператор фискальных данных) →
        Чек напечатан + фискальный признак получен →
          Сохранить fiscal_number в заказе →
            closeOrder() на сервере
```

### ESC/POS протокол (нефискальные чеки)

- `escpos.ts` — билдер команд: инициализация, выравнивание, жирный, двойная высота, разделители, отрезка бумаги
- Кодировка **CP866** для русского текста
- Ширина: 32 символа (58мм) или 48 символов (80мм)

### Подключение

| Устройство | Способ | Платформа | Библиотека |
|---|---|---|---|
| Термопринтер (пре-чеки) | Bluetooth BLE | Android + iOS | `react-native-ble-plx` |
| Термопринтер (пре-чеки) | USB OTG | Только Android | Кастомный native-модуль |
| ККТ АТОЛ (фискальные) | Bluetooth / USB | Android + iOS | ATOL SDK (native module) |

## Критерии приёмки

1. **Фаза 1:** PIN-логин → открыть смену → создать заказ на стол → добавить товары → оплатить → закрыть смену
2. **Фаза 2:** Отключить WiFi → создать заказ → добавить товары → оплатить → включить WiFi → заказ появился в веб-версии
3. **Фаза 3 (печать):** Подключить Bluetooth-термопринтер → закрыть заказ → пре-чек напечатан
4. **Фаза 3 (ККТ):** Подключить АТОЛ по Bluetooth → закрыть заказ → фискальный чек напечатан → fiscal_number сохранён на сервере
5. **Фаза 4:** Установить из App Store/Google Play → полный цикл работы кассира
